Pages are composed from many independently-authored components. These practices keep one
component's CSS from breaking another's — and keep pages fast.

## Scope styles to the component

<Callout type="dont">Bare element selectors leak across the whole page: `h2 { color: red; }`, `ul { margin: 0; }`, `a { … }`. Any other component's `h2`/`ul`/`a` is now affected.</Callout>
<Callout type="do">Namespace everything under the component's block class: `.teaser h2` or, better, `.teaser__title`. A component should only ever style **its own** subtree.</Callout>

## Keep specificity low and flat

- One class per element (BEM). Avoid IDs and long descendant chains.
- Never reach for `!important` to win a fight — fix the selector or use a modifier class. (If you
  must override a library, prefer **cascade layers** — see the CSS track.)

## Manage clientlib relationships

Two properties on a clientlib (in its `.content.xml`) say how categories relate. They sound similar but
do very different things:

- **`dependencies`** — "load that category **before** me, as its **own** file." Load order is guaranteed; the two libs stay **separate** requests.
- **`embed`** — "**copy** that category's compiled output **inside** mine." One request; the other lib's code is **duplicated** into my file.

### See the difference in the page output

A `card` clientlib that needs the atomic `colors`:

**With `dependencies`** → two files, loaded in order (colors stays shared & cacheable):

```html
<link rel="stylesheet" href="/etc.clientlibs/academy/.../colors.css">  <!-- loads first -->
<link rel="stylesheet" href="/etc.clientlibs/academy/.../card.css">    <!-- then this    -->
```

**With `embed`** → one file; the colors are baked into it:

```html
<link rel="stylesheet" href="/etc.clientlibs/academy/.../card.css">    <!-- colors + card, inlined -->
```

### Which to use

```text
# dependencies — shared base kept as ONE cached file, just ordered first
dependencies=[academy.publish.atomic.colors]

# embed — bundle a few small libs into ONE request (e.g. a site / proxy clientlib)
embed=[academy.publish.atomic.colors, academy.publish.atomic.spacing]
```

<Callout type="do">✅ **Good — `dependencies` for anything shared** (colors, base libs). It's downloaded **once** and the browser **caches that single file** across every page and component that depends on it.</Callout>

<Callout type="warn">❌ **Bad — `embed`-ing a shared base into every component.** Each component then carries its **own copy** of those styles, so the same CSS ships again and again with no shared cache. Reserve `embed` for **combining a few small libs into one request**, never for sharing.</Callout>

## Performance (ties into the Web Performance track)

- Ship only what a page needs — per-component clientlibs beat one giant global stylesheet.
- CSS is **render-blocking**: keep it lean, minified (the build does this), and consider
  **critical CSS** for above-the-fold content.
- Style media with `width`/`height` or `aspect-ratio` to avoid layout shift (CLS).
- Prefer animating `transform`/`opacity` (compositor-friendly) over layout properties.

## Theming & variants

- Expose brand/spacing as **CSS custom properties** so themes and the **Style System** can override
  them live, without recompiling.
- Author-selectable variants = BEM **modifier** classes toggled by `cq:styleGroups` (Components
  track) — no per-variant code.

## Responsive

- Mobile-first: base styles for small screens, layer up with `min-width` media queries using the
  **shared breakpoints** (`common/styles`).
- Lean on intrinsic sizing (Flexbox/Grid wrapping, `min()`/`max()`/`clamp()`) so a component adapts
  to the space it's given with as few media queries as possible.

## Quick checklist

<Callout type="do">Before you commit a component's SCSS, check: scoped to the block class · BEM, low specificity · no `!important` · uses shared tokens, not hard-coded values · author-only styles behind `wcmmode` · media sized to avoid CLS · passes stylelint.</Callout>
