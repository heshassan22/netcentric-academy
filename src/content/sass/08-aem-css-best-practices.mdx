Pages are composed from many independently-authored components. These practices keep one
component's CSS from breaking another's ŌĆö and keep pages fast.

## Scope styles to the component

<Callout type="dont">Bare element selectors leak across the whole page: `h2 { color: red; }`, `ul { margin: 0; }`, `a { ŌĆ” }`. Any other component's `h2`/`ul`/`a` is now affected.</Callout>
<Callout type="do">Namespace everything under the component's block class: `.teaser h2` or, better, `.teaser__title`. A component should only ever style **its own** subtree.</Callout>

## Keep specificity low and flat

- One class per element (BEM). Avoid IDs and long descendant chains.
- Never reach for `!important` to win a fight ŌĆö fix the selector or use a modifier class. (If you
  must override a library, prefer **cascade layers** ŌĆö see the CSS track.)

## Manage clientlib relationships

Two properties on a clientlib (in its `.content.xml`) say how categories relate. They sound similar but
do very different things:

- **`dependencies`** ŌĆö "load that category **before** me, as its **own** file." Load order is guaranteed; the two libs stay **separate** requests.
- **`embed`** ŌĆö "**copy** that category's compiled output **inside** mine." One request; the other lib's code is **duplicated** into my file.

### See the difference in the page output

A `card` clientlib that needs the atomic `colors`:

**With `dependencies`** ŌåÆ two files, loaded in order (colors stays shared & cacheable):

```html
<link rel="stylesheet" href="/etc.clientlibs/academy/.../colors.css">  <!-- loads first -->
<link rel="stylesheet" href="/etc.clientlibs/academy/.../card.css">    <!-- then this    -->
```

**With `embed`** ŌåÆ one file; the colors are baked into it:

```html
<link rel="stylesheet" href="/etc.clientlibs/academy/.../card.css">    <!-- colors + card, inlined -->
```

### Which to use

```text
# dependencies ŌĆö shared base kept as ONE cached file, just ordered first
dependencies=[academy.publish.atomic.colors]

# embed ŌĆö bundle a few small libs into ONE request (e.g. a site / proxy clientlib)
embed=[academy.publish.atomic.colors, academy.publish.atomic.spacing]
```

<Callout type="do">Ō£ģ **Good ŌĆö `dependencies` for anything shared** (colors, base libs). It's downloaded **once** and the browser **caches that single file** across every page and component that depends on it.</Callout>

<Callout type="warn">ŌØī **Bad ŌĆö `embed`-ing a shared base into every component.** Each component then carries its **own copy** of those styles, so the same CSS ships again and again with no shared cache. Reserve `embed` for **combining a few small libs into one request**, never for sharing.</Callout>

## Troubleshooting: my SCSS change isn't in the CSS

SCSS is a **build input** ŌĆö AEM never serves your `.scss` directly. The front-end build compiles it
into the clientlib's CSS, and the *compiled* file is what loads. So when an edit "doesn't show up", the
new CSS almost always wasn't **produced** or wasn't **deployed**. Check these, most common first:

1. **You didn't rebuild + redeploy.** Editing `.scss` does nothing until the FE build runs. Rebuild and install the front-end module: `mvn clean install -PautoInstallPackage -pl ui.frontend`. Until then AEM keeps serving the previously compiled CSS.
2. **The build failed, so no new file was written.** A SCSS compile error ŌĆö or a **stylelint** error ŌĆö aborts the bundle, and AEM keeps the last good CSS. Read the build log, fix the error, rebuild. (A green "BUILD SUCCESS" line is not enough ŌĆö scroll up for stylelint warnings/errors.)
3. **The partial isn't imported.** A new `_card.scss` only compiles if an entry file `@use`s / `@imports` it. An orphan partial is silently ignored ŌĆö no error, no output.
4. **You edited the compiled output, not the source.** Generated CSS (under `dist` / the clientlib output) is **overwritten on every build**. Edit the `.scss` source, never the built `.css`.
5. **Something is caching the old file.** Hard-refresh the browser (it caches the hashed clientlib URL); rebuild AEM's clientlibs via `/libs/granite/ui/content/dumplibs.rebuild.html`; and on **publish**, the **dispatcher** may serve a cached copy ŌĆö flush it.
6. **The clientlib category isn't on the page.** The CSS compiled fine, but that category isn't included (HTL clientlib include / page policy), so you never see it. Confirm the `<link>` is actually in the page source.

<Callout type="do">The golden rule: **edit the `.scss`, rebuild, redeploy, hard-refresh.** If the change still isn't there, the build almost certainly **failed** (check the log) or you're editing the **compiled output** instead of the source.</Callout>

## Performance (ties into the Web Performance track)

- Ship only what a page needs ŌĆö per-component clientlibs beat one giant global stylesheet.
- CSS is **render-blocking**: keep it lean, minified (the build does this), and consider
  **critical CSS** for above-the-fold content.
- Style media with `width`/`height` or `aspect-ratio` to avoid layout shift (CLS).
- Prefer animating `transform`/`opacity` (compositor-friendly) over layout properties.

## Theming & variants

- Expose brand/spacing as **CSS custom properties** so themes and the **Style System** can override
  them live, without recompiling.
- Author-selectable variants = BEM **modifier** classes toggled by `cq:styleGroups` (Components
  track) ŌĆö no per-variant code.

## Responsive

- Mobile-first: base styles for small screens, layer up with `min-width` media queries using the
  **shared breakpoints** (`common/styles`).
- Lean on intrinsic sizing (Flexbox/Grid wrapping, `min()`/`max()`/`clamp()`) so a component adapts
  to the space it's given with as few media queries as possible.

## Quick checklist

<Callout type="do">Before you commit a component's SCSS, check: scoped to the block class ┬Ę BEM, low specificity ┬Ę no `!important` ┬Ę uses shared tokens, not hard-coded values ┬Ę author-only styles behind `wcmmode` ┬Ę media sized to avoid CLS ┬Ę passes stylelint.</Callout>
