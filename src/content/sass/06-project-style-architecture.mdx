A maintainable AEM front end shares **one source of truth** for design decisions — colours,
breakpoints, spacing, fonts, z-index — and every component imports from it instead of hard-coding
values.

## The shared `common/styles` layer

```text
ui.frontend/src/main/frontend/academy/common/styles/
├── variables/
│   ├── colors.scss
│   ├── breakpoints.scss
│   ├── fonts.scss
│   ├── grid.scss
│   └── z_index.scss
└── variables.scss        ← an entry point that @forwards the partials
```

The build's SASS `includePaths` point at `common/styles`, so any component can `@use` these without
long relative paths:

```scss
@use "variables/colors" as *;
@use "variables/breakpoints" as bp;

.teaser__title {
  color: $brand;
  @media (min-width: bp.$tablet) { font-size: 2.5rem; }
}
```

## `@use` and `@forward` (modern Sass)

- **`@use "file"`** imports a module **once**, namespaced (`bp.$tablet`) — no global leakage, no
  duplicate output. `@use "file" as *` drops the namespace.
- **`@forward "file"`** re-exports a module so an **entry point** (like `_index.scss`) can pass it
  along to whatever imports that entry point.

<Callout type="note">Rule of thumb: **`@use`** a file when you need its variables/mixins **in the current file**. **`@forward`** a file only when you're building an entry point (like `_index.scss`) and want to **pass it along** to other files — without using it there yourself.</Callout>

<Callout type="warn">`@import` is **deprecated** in Sass — it re-includes files (duplication) and dumps everything into the global scope. Use `@use`/`@forward` in new code.</Callout>

## Design tokens → Sass vs CSS custom properties

Two kinds of "variable", and they're complementary:

| | Sass `$var` | CSS `--var` |
|---|---|---|
| Resolved | at **build time** | **live** in the browser |
| Good for | breakpoints, maps, build-time math | theming, runtime changes, per-component overrides |
| Cascades/inherits | no | yes |

A common pattern: define brand tokens as **CSS custom properties** (so they can be themed/overridden
live), and keep build-time concerns (breakpoints, maps) as **Sass variables**:

```scss
:root {
  --brand: #6b2fb3;
  --space: 1rem;
}
.teaser { padding: var(--space); border-color: var(--brand); }
```

## Theming with custom properties

Because custom properties are live, a single switch re-themes everything — e.g. a brand/section
theme set by a class the author picks (think AEM **Style System**, see the Components track):

```scss
.theme--dark { --brand: #b794ec; --bg: #14141c; }
```

Toggle the theme and watch every value follow — **Open in CodePen** gives you a runnable copy:
