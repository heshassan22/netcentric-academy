Classes (lesson 02) are *one* way to structure a component. But most component behaviour in this
codebase is written **functionally** — module-scoped functions and **closures**. It's simpler and the
style you'll meet most often in the project.

## Same job, two styles

A component does the same three things either way: find its elements, hold a little state, react to
events. The difference is *how* you package that.

| | Class (lesson 02) | Functional (this lesson) |
|---|---|---|
| create it | `new Widget(el)` | `init(el)` |
| hold state | `this.count` on the instance | a `count` variable in the **closure** |
| behaviour | `this.method()` | inner functions |
| best for | modelling entities, inheritance | most DOM / UI behaviour |

## The shape: an IIFE module

A component file is wrapped in an **IIFE** — an Immediately Invoked Function Expression. It runs the
moment the file loads and creates a **private scope**, so nothing leaks to the global `window`.

```js
(() => {
  'use strict';

  // everything lives in here — private to this module

})();
```

## Selector constants

Collect the CSS selectors in one object at the top. The JS, HTML and SCSS all share these class
names — keeping them in one place makes a rename a one-line change.

```js
const SELECTORS = {
  root: '.stepper',
  value: '.stepper__value',
  dec: '.stepper__btn--dec',
  inc: '.stepper__btn--inc',
};
```

## `init(element)` — closures instead of `this`

Instead of a constructor storing state on `this`, an **`init` function** receives the element and
keeps everything in local variables. The event handlers **close over** those variables — that's the
closure doing the job `this` did in the class version.

```js
const init = (root) => {
  const input = root.querySelector(SELECTORS.value);
  const dec   = root.querySelector(SELECTORS.dec);
  const inc   = root.querySelector(SELECTORS.inc);
  if (!input || !dec || !inc) return;        // defensive guard — bail if markup is missing

  const step = Number(input.step) || 1;
  const min  = Number(input.min);
  const max  = Number(input.max);

  // `apply` closes over input/dec/inc/min/max
  const apply = (delta) => {
    input.value  = Math.min(Math.max(Number(input.value) + delta, min), max);
    dec.disabled = Number(input.value) <= min;
    inc.disabled = Number(input.value) >= max;
  };

  dec.addEventListener('click', () => apply(-step));
  inc.addEventListener('click', () => apply(step));
  apply(0);                                   // set the initial disabled state
};
```

Each call to `init(root)` makes a **fresh closure** with its own `input`, `min`, `max` — so two
steppers on a page stay independent, exactly like two class instances did.

## `initAll()` — one per element on the page

Find every root element and initialise each:

```js
const initAll = () => document.querySelectorAll(SELECTORS.root).forEach(init);

document.addEventListener('DOMContentLoaded', initAll);
```

## State lives in the closure

Handlers are arrow functions that close over the local variables, so they always have the element and
state they need — the callback context problem from lesson 02 never comes up.

## How it loads in AEM

Each component is a small JS file bundled by webpack into its **own clientlib**. The file
is an IIFE that **self-initialises** — it runs `initAll()` on `DOMContentLoaded`, finds its elements,
and wires them up — it self-initialises. (Expose `initAll` so it can run again if the DOM is re-rendered.)

<Callout type="do">Reach for the **functional** style for everyday component behaviour: an IIFE, selector constants, an `init(element)` that closes over its state, and an `initAll()` that runs it per element. It's the project's default.</Callout>

<Callout type="note">Use a **class** when you genuinely model an entity with inheritance, or when an API hands you instances. For DOM behaviour, functions + closures are usually clearer.</Callout>
