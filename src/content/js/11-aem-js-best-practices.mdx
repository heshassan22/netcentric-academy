The habits that make component JS robust on author-composed pages — a checklist for everything in
this track.

<Callout type="note">**In the example** (`examples/js-component`): the whole `accordion.clientlibs.js` is these habits in practice — `[11]` exports the class for unit testing; scoping, the `data-*` contract, and no globals run throughout.</Callout>

## Scope to `this.element`

<Callout type="dont">`document.querySelector('.button')` — grabs the first match anywhere, so it hits the wrong instance when a page has several of your component.</Callout>
<Callout type="do">`this.element.querySelector('.button')` — only ever touch your own subtree. Many instances coexist safely.</Callout>

## Progressive enhancement

The HTML should be useful **before** JS runs (and if it fails). JS *enhances*; it isn't the only way
to get content.

```html
<!-- works as a link with no JS; the component enhances it -->
<a class="accordion__trigger" href="#panel-1" data-nc="Accordion">Details</a>
```

<Callout type="do">Render real content/links in HTL; use JS to upgrade behaviour. Provide `<noscript>` fallbacks for JS-only features (e.g. a lazy-loaded image), as in the lazy-loading exercise.</Callout>

## The `data-*` contract

`data-nc-params-<Component>` (valid JSON) is the clean channel from HTL/back-end to JS. Don't scrape
config out of class names or text content.

```html
<div data-nc="Gallery" data-nc-params-Gallery='{"autoplay":true,"interval":5000}'></div>
```

```js
const { autoplay, interval } = this.options;
```

## No global state, no leaks

- Keep state on the instance (`this.x`), never on `window`.
- Don't pollute globals; rely on ES module scope.
- Make `init()` **idempotent** and keep listener references so you can remove them (author re-init —
  see lesson 09).

## Keep it dependency-light

<Callout type="warn">Avoid jQuery and heavy libraries for things the platform now does natively (`querySelector`, `classList`, `fetch`, `IntersectionObserver`). Every KB ships to every visitor — see Web Performance.</Callout>

## Write testable code

Pure logic in **modules** (tested with Jest), thin components that wire modules to the DOM. A
component that takes `(element, options)` is easy to unit-test in jsdom — see the Unit Testing
track's DOM lesson.

## Accessibility & resilience

- Manage focus and ARIA state in JS widgets (e.g. `aria-expanded` on a toggle).
- Guard DOM access: `const btn = this.element.querySelector('.btn'); if (!btn) return;` — markup may
  vary by authoring.

## Commit checklist

<Callout type="do">Before committing component JS: queries scoped to `this.element` · `init()` idempotent · listeners removable · config via `data-nc-params` (valid JSON) · no globals · works (or degrades) without JS · heavy/rare code dynamically imported · pure logic extracted &amp; unit-tested · no stray `console.log`.</Callout>

<Callout type="note">Together with the loader lessons (03–06), this is how the project keeps component JS predictable across hundreds of author-composed pages.</Callout>
