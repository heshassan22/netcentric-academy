When the same logic shows up in several components, extract it into a **module** and import it.
Shared, tested utilities beat copy-pasted snippets.

<Callout type="note">**In the example** (`examples/js-component`): `accordion.clientlibs.js` `[08]` imports `common/scripts/dom.js` (`qsa`, `emit`) instead of re-writing them.</Callout>

## A shared module

Put cross-component helpers in a common folder — e.g. an IntersectionObserver wrapper used by any
component that lazy-initialises:

```js
// publish/modules/intersection-observer.js
export default function observe(target, callback, once = false) {
  const io = new IntersectionObserver((entries, obs) => {
    entries.forEach((entry) => {
      if (entry.isIntersecting) {
        callback(entry);
        if (once) obs.unobserve(target);
      }
    });
  });
  io.observe(target);
  return io;
}
```

```js
// in a component
import observe from '../../modules/intersection-observer.js';

init() {
  observe(this.element, () => this.loadImage(), true);
}
```

## Named vs default exports

```js
// utils/format.js
export function formatPrice(cents) { /* … */ }   // named — many per file
export function slugify(text) { /* … */ }

export default class Carousel { /* … */ }         // default — one main thing
```

```js
import Carousel, { formatPrice, slugify } from './…';
```

<Callout type="do">Prefer **named exports** for utility collections (better autocomplete and refactoring); use a **default export** for a module's single main thing (a component class).</Callout>

## Keep modules small and pure

- A utility module should do **one thing** and avoid side effects (no touching the DOM at import
  time, no global state).
- Pure functions (input → output, no side effects) are trivial to **unit test** — see the Unit
  Testing track.

```js
// pure + testable
export const clamp = (n, min, max) => Math.min(Math.max(n, min), max);
```

## Suggested structure

```text
academy/publish/
├── components/        # one folder per component (JS + SCSS)
│   ├── teaser/
│   └── my-component/
├── modules/           # shared, reusable logic (observers, fetch helpers…)
└── utils/             # small pure helpers (format, dom, math)
```

<Callout type="warn">Watch for circular imports (A imports B imports A) — they cause `undefined` exports at runtime. If two modules need each other, the shared piece usually belongs in a third module.</Callout>

<Callout type="note">Modules + the component pattern are what keep a growing AEM front end maintainable: thin components that compose small, tested modules.</Callout>
