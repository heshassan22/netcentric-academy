You read tests far more than you write them, so keep them **easy to find and easy to read**.

## Where tests live

Jest finds both of these — pick one and be consistent:

```text
src/utils/format.js
src/utils/format.test.js          # right next to the file (recommended)

src/utils/__tests__/format.test.js   # or in a __tests__ folder
```

Putting the test **next to the file** keeps it visible and you update both in the same change.

## Group with `describe`

`describe` boxes group related tests; nest them per situation so the output reads like a checklist:

```js
describe('applyDiscount()', () => {
  describe('with a normal percent', () => {
    it('takes the right amount off', () => { /* … */ });
  });
  describe('with a bad percent', () => {
    it('throws on a negative percent', () => { /* … */ });
  });
});
```

## Name tests like sentences

The name should say the **behaviour**, so a red test reads like a spec:

<Callout type="dont">`test('test1')` · `test('it works')` — tells you nothing when it fails.</Callout>
<Callout type="do">`it('takes 10% off the total')` — you know exactly what broke before you even open the code.</Callout>

## Reuse setup with a small helper

If many tests need a similar object, a tiny **builder** keeps them short — and lets each test show only
what it cares about:

```js
function makeUser(overrides = {}) {
  return { id: 1, name: 'Ada', active: true, ...overrides };
}

test('inactive users are hidden', () => {
  expect(isVisible(makeUser({ active: false }))).toBe(false);   // only 'active' matters here
});
```

<Callout type="do">Don't over-engineer. If you have to scroll away to understand a test, a little repetition is better. In tests, **readability beats DRY** — the reader should see arrange → act → assert in one glance.</Callout>
