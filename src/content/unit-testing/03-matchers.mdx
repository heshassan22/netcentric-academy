A **matcher** is the check you write after `expect(...)`. Picking the right one gives you a clear
failure message instead of a useless "expected false".

## Checking values: `toBe` vs `toEqual`

```js
expect(2 + 2).toBe(4);                  // toBe → for numbers, strings, booleans
expect({ name: 'Ada' }).toEqual({ name: 'Ada' });   // toEqual → for objects & arrays
```

Why two? `toBe` asks *"is it the exact same value?"*. For objects that means *the same object in
memory* — so two objects that merely **look** identical are **not** `toBe` equal:

<Callout type="warn">`expect({ a: 1 }).toBe({ a: 1 })` **fails** — they're two different objects. Use **`toEqual`** for objects and arrays (it compares the contents), and **`toBe`** for plain values.</Callout>

## Numbers & strings

```js
expect(price).toBeGreaterThan(0);
expect(0.1 + 0.2).toBeCloseTo(0.3);     // floats are fuzzy — never toBe them
expect('Netcentric').toMatch(/centric/); // contains (regex or substring)
```

## Arrays & objects

```js
expect(['a', 'b', 'c']).toContain('b');           // is it in the array?
expect(items).toHaveLength(3);                      // how many?
expect(user).toHaveProperty('name', 'Ada');         // a specific property
expect(user).toMatchObject({ name: 'Ada' });        // has at least these fields
```

## Errors

When you expect code to **throw**, pass a **function** so `expect` can run it and catch the error:

```js
expect(() => parse('')).toThrow();                  // throws something
expect(() => parse('')).toThrow('empty input');     // message contains this
```

<Callout type="note">Pass `() => parse('')`, not `parse('')`. If you write `expect(parse('')).toThrow()`, the error happens *before* `expect` runs and the test crashes instead of checking.</Callout>

## DOM matchers

With `@testing-library/jest-dom` you get readable checks for DOM elements (more in the DOM lesson):

```js
expect(button).toBeInTheDocument();
expect(button).toBeDisabled();
expect(el).toHaveTextContent('Subscribe');
```

<Callout type="do">Reach for the **most specific** matcher. `toHaveLength(3)` reads (and fails) more clearly than `toBe(3)` on `list.length`; `toBeInTheDocument()` is clearer than `not.toBeNull()`.</Callout>
