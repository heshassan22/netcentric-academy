The last piece: knowing **how much** of your code the tests actually run, and making sure those tests
**mean something** — not just chasing a number.

## See your coverage

```bash
npx jest --coverage
```

Jest prints four numbers:

| Metric | Plain meaning |
|---|---|
| **Statements** | how many lines of code ran |
| **Branches** | how many `if`/`else` paths were taken |
| **Functions** | how many functions were called |
| **Lines** | (almost the same as statements) |

**Branches** is the one to watch — it's easy to run an `if` but only ever test the *true* side.

## Make the build enforce a minimum

```js
// jest.config.js
export default {
  collectCoverage: true,
  coverageThreshold: {
    global: { branches: 80, functions: 80, lines: 80, statements: 80 },
  },
};
```

Now CI fails if coverage drops below 80%.

## Coverage is a guide, not a goal

A line can be *run* by a test that doesn't actually check anything — that counts as "covered" but
proves nothing:

```js
// 100% coverage, 0% value — it runs the code but asserts nothing
test('renders', () => { render(); });
```

<Callout type="warn">100% coverage does **not** mean bug-free. Coverage tells you what *isn't* tested — it can't tell you whether what *is* tested is tested well.</Callout>

## What's actually worth testing

<Callout type="do">Spend your effort on: <strong>logic</strong> (calculations, formatting, validation), <strong>edge cases</strong> (empty, null, zero, the error path), and <strong>behaviour</strong> (what happens on a click). That's where bugs hide.</Callout>
<Callout type="dont">Don't chase coverage on trivial getters or config, don't test the framework or the language, and don't assert on internals that change every refactor.</Callout>

<Callout type="note">Quick gut-check: break a line of real code on purpose. If **no test goes red**, that line isn't meaningfully tested — no matter what the coverage number says.</Callout>
