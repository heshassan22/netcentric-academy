Build the **Blog Explorer** as real AEM components — the same app as the JS track's
[final project](#/js/final-project), assembled the AEM way. It puts the whole module to work:
structure, dialog, Sling Model, clientlibs (functional loader JS), policies, and the Style System.
Build it in the **`code`** AEM project.

Data comes from [JSONPlaceholder](https://jsonplaceholder.typicode.com/): `/posts`, `/users`,
`/posts/{id}/comments` — fetched **client-side** from the component JS (its CORS is open).

Click a card  →  Post Dialog (<dialog>) with the post's body + comments

Responsive: 1 column on mobile, 2–3 as it widens (SASS + a breakpoint mixin).

## The component and its three modules

Build **one** AEM component — **Blog Explorer** (`apps/academy/components/blog-explorer`) — whose
HTL renders the header/search, the grid, and the dialog. Its clientlib JS composes three small
**functional modules** (the same split as the JS project) and wires them — mostly by **callbacks**,
with **one** custom event. The entry JS is the AEM equivalent of `main.js`.

1. **search-bar** — `createSearchBar(root, { onSearch })`: a `<form>` with a search input; on input
   (**debounced**) it calls `onSearch(query)`.
2. **post-list** — `createPostList(root)` returns **`{ render(posts), filter(query) }`**. It renders
   the `<article>` cards, **fades them in** with an IntersectionObserver, and filters by query. When
   a card is clicked it dispatches a single **`post:open`** `CustomEvent` (bubbling, the post in
   `detail`).
3. **post-dialog** — `createPostDialog(root)` **listens for `post:open`**, fetches that post's
   comments, and shows the title, body, and comments in a `<dialog>`, with loading and error states.

## The Sling Model (provided)

You focus on the front end — here's the model to drop into `core`. It reads the dialog values
(`heading`, `placeholder`, `count`); your HTL passes `count` to the JS via a `data-count` attribute.

```java
package biz.netcentric.academy.core.models.blogexplorer;

import org.apache.sling.api.resource.Resource;
import org.apache.sling.models.annotations.Default;
import org.apache.sling.models.annotations.DefaultInjectionStrategy;
import org.apache.sling.models.annotations.Model;
import org.apache.sling.models.annotations.injectorspecific.ValueMapValue;

@Model(adaptables = Resource.class, defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class BlogExplorerModel {

    @ValueMapValue @Default(values = "Blog Explorer")
    private String heading;

    @ValueMapValue @Default(values = "Search posts…")
    private String placeholder;

    @ValueMapValue @Default(intValues = 12)
    private int count;

    public String getHeading() { return heading; }
    public String getPlaceholder() { return placeholder; }
    public int getCount() { return count; }
}
```

## Requirements

- [ ] **Structure** — one `blog-explorer` component rendered by its HTL (header/search + grid + `<dialog>`)
- [ ] **Dialog** — authorable options: the heading, the search placeholder, the post count
- [ ] **Sling Model** — read those dialog values and keep HTL a thin view
- [ ] **Clientlibs** — BEM SCSS (tokens + a breakpoint mixin for the responsive grid), loaded by category
- [ ] **Functional JS** — `register` a **function** (not a class); compose three functional modules (search-bar, post-list, post-dialog) wired by callbacks; fetch, transform with pure helpers (join authors, build excerpts), render, **debounce**, and **lazy-reveal** the cards
- [ ] **One custom event** — a card click dispatches `post:open` and the dialog listens; nothing else uses events
- [ ] **Loading & error states** on every fetch
- [ ] **Semantic HTML** in HTL — `<header>`, `<main>`, `<article>`, `<form>`, `<dialog>`
- [ ] **Policy + Style System** — allow the component on the template, and add at least one Style System variation (e.g. grid density, or a card style)

How you split and name the JS functions is up to you — aim for clean, readable modules.

<Callout type="note">Fetch **client-side** from the component JS to keep it simple. As a stretch, move the fetch server-side into a **Sling Model or servlet** (so the markup ships rendered and the calls can be cached) — that's how a production AEM build would usually do it.</Callout>

## Acceptance

<Callout type="do">Done when: posts load (each card showing its author), the search filters live, cards fade in on scroll, clicking a card opens a dialog with that post's comments, and every fetch shows loading / handles errors — built as real AEM components (HTL + dialog + Sling Model + clientlibs), wired with **functional** loader JS and **one** custom event, allowed by policy with a Style System variation.</Callout>
