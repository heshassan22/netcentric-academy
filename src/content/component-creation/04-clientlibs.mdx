A component usually needs CSS (and sometimes JS). AEM delivers these through a **client library**
(clientlib) — a folder of CSS/JS that AEM bundles, minifies, and serves under a **category**
name you reference from HTL.

## Where styles come from in this project

The front end lives in `ui.frontend`. You author SCSS per component:

```text
ui.frontend/src/main/frontend/academy/publish/components/hello/
└── hello.clientlibs.scss
```

The build (`nc-fe-build`) compiles it into a generated clientlib whose **category** follows the
folder path: `academy.publish.components.hello`.

<Callout type="note">A clientlib is generated **only for components that have a source file here**. A markup-only component (like the `hello` from earlier lessons, with no `.clientlibs.scss`/`.js`) produces **no clientlib** — that's expected, not an error. Add a source file only when the component actually needs styling or behaviour.</Callout>

## Loading the clientlib from HTL

Pull the category in with the Granite `clientlib` template:

```html
<sly data-sly-use.clientlib="${'/libs/granite/sightly/templates/clientlib.html'}"
     data-sly-call="${clientlib.all @ categories='academy.publish.components.hello'}"/>

<h2 class="hello__title">${hello.greeting}</h2>
```

- `clientlib.all` injects both CSS and JS. There are also `clientlib.css` and `clientlib.js`
  if you need only one.
- The `categories` string must match the generated category exactly.

## Author mode & the decoration tag

A component is **edited in author mode** and **viewed in publish** — the DOM and the needs differ.

- **The decoration tag** — AEM usually wraps your component in a `<div>` (the *decoration tag*) carrying its CSS class and edit hooks, so your markup isn't always the outermost element. Style your **own** root class (`.hello`), not "the wrapper". Control it per component in `.content.xml` with `cq:noDecoration="{Boolean}true"` (no wrapper) or `cq:htmlTag`. The **Style System** adds the author-selected class to this tag automatically (see *Policies*).
- **Author-only styling** — load edit affordances (outlines, "empty" hints) behind a `wcmmode` guard so they never ship to publish:

```html
<sly data-sly-test="${wcmmode.edit}"
     data-sly-use.clientlib="${'/libs/granite/sightly/templates/clientlib.html'}"
     data-sly-call="${clientlib.all @ categories='academy.author.components.hello'}"/>
```

The matching SCSS lives under `academy/author/…`, so the build emits an `academy.author.*` category loaded only in edit mode.

<Callout type="warn">In author mode your component sits inside editor chrome (overlays, the decoration wrapper). Don't assume it's the outermost element or rely on `:first-child`/`:last-child` against the page.</Callout>

## Adding JavaScript — the component loader

CSS is declarative; for **behaviour** you add a JS file beside the component and wire it up with the
[Netcentric component loader](https://github.com/Netcentric/component-loader). You never call your
code directly — you **register** a class, and the loader creates an instance for every matching
element on the page (including ones authors add live).

### 1 · Write the component's JS

The source sits next to the SCSS:

```text
ui.frontend/src/main/frontend/academy/publish/components/hello/
├── hello.clientlibs.scss
└── hello.clientlibs.js
```

```js
import { register } from '@netcentric/component-loader';

class Hello {
  constructor(element, options) {
    this.element = element;   // the DOM node carrying data-nc="Hello"
    this.options = options;   // parsed from data-nc-params-Hello (JSON)
    this.init();
  }

  init() {
    // Scope queries to this.element — never document.querySelector.
    this.element.addEventListener('click', () => {
      this.element.classList.toggle('hello--active');
    });
  }
}

register({ Hello });   // the key "Hello" is the name you use in data-nc
export default Hello;
```

### 2 · Make the build aware of it

Export it from the publish entry so the bundler includes it — if it's never imported, it's never
registered and nothing runs:

```js
// publish/components/index.js
export * from 'academy-base-project/publish/components/hello/hello';
```

### 3 · The loader runtime (set up once per page)

One entry boots the loader: `run()` initialises components already in the DOM, and `observe()`
initialises any added afterwards (essential for author mode):

```js
// publish.clientlibs.js
import { observe, run } from '@netcentric/component-loader';
run();        // defaults to scanning document for the data-nc attribute
observe();
```

### 4 · Wire it from HTL

Mark the element with `data-nc` (the **registered name** — case-sensitive). Optionally pass config
as **valid JSON** in `data-nc-params-<Name>`; it arrives as the constructor's `options`:

```html
<sly data-sly-use.clientlib="${'/libs/granite/sightly/templates/clientlib.html'}"
     data-sly-call="${clientlib.all @ categories='academy.publish.components.hello'}"/>

<div class="hello" data-nc="Hello" data-nc-params-Hello='{"speed":300}'>
  <h2 class="hello__title">${hello.greeting}</h2>
</div>
```

### Loader features worth knowing

- **Multiple components on one element:** `data-nc="Hello,Tracker"` (+ a `data-nc-params-…` per name).
- **Lazy init:** `data-nc-loading="lazy"` defers instantiation until the element scrolls into view — ideal for heavy or below-the-fold components.
- **Introspection:** the loader exports `components` (registered classes) and `instances` (live instances by name).

<Callout type="warn">The loader ships **untranspiled**, so the front-end build must run Babel over it — include `@netcentric` in `babel-loader` rather than blanket-excluding `node_modules`. `nc-fe-build` already handles this; it only bites in a hand-rolled webpack config.</Callout>

<Callout type="note">This is the component-author's view. For the behaviour itself — events, delegation, custom events, params/DOM references, and performance patterns — see the **JavaScript track**.</Callout>

## The manual alternative — `cq:ClientLibraryFolder`

Without the FE build you can define a clientlib node by hand under the component (or under
`/apps/academy/clientlibs`):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<jcr:root xmlns:cq="http://www.day.com/jcr/cq/1.0" xmlns:jcr="http://www.jcp.org/jcr/1.0"
    jcr:primaryType="cq:ClientLibraryFolder"
    allowProxy="{Boolean}true"
    categories="[academy.publish.components.hello]"/>
```

with sibling files listing what to include:

```text
hello-clientlib/
├── .content.xml      ← cq:ClientLibraryFolder (above)
├── css.txt           ← #base=css  →  hello.css
├── js.txt            ← #base=js   →  hello.js
├── css/hello.css
└── js/hello.js
```

The FE build just automates generating this from your SCSS/JS sources.

## Useful clientlib properties

| Property | Purpose |
|---|---|
| `categories` | the name(s) you load by |
| `allowProxy` | serve under `/etc.clientlibs` (recommended, keeps `/apps` private) |
| `dependencies` | other categories that must load first |
| `embed` | inline another category's output into this one |

<Callout type="do">Name categories by environment + type + component (`academy.publish.components.hello`). Consistent naming keeps the clientlib graph predictable as the project grows.</Callout>
