A graded set of assignments covering the whole module. Each level builds on the one before, so
do them in order — by the end you have one component that an author can insert, fill, style, and
configure with no code changes.

<Callout type="note">The **Java (Sling Models) is provided** for every task that needs it — drop each into `core`, then rebuild & deploy: `mvn clean install -PautoInstallSinglePackage`. Your focus is the **front end**: HTL, dialog, clientlibs, SCSS/JS, policy, and the Style System. Place components under a page's `root/responsivegrid` (not directly under `root`).</Callout>

### 1 — Create and render a component

- [ ] Create `apps/academy/components/notice/` with a `.content.xml` node of type `cq:Component`, a `jcr:title`, and `componentGroup="Academy"`
- [ ] Add `notice.html` that renders static markup: `<div class="notice">A static notice.</div>`
- [ ] Deploy, place the component under `root/responsivegrid`, and confirm it renders

**Acceptance:** the component renders on a page from its own `notice.html` script.

### 2 — Render derived content (no dialog, no author input)

Java (provided — `core/.../models/pagemeta/PageMetaModel.java`):

```java
package biz.netcentric.academy.core.models.pagemeta;

import com.day.cq.wcm.api.Page;
import org.apache.sling.api.SlingHttpServletRequest;
import org.apache.sling.models.annotations.DefaultInjectionStrategy;
import org.apache.sling.models.annotations.Model;
import org.apache.sling.models.annotations.injectorspecific.ScriptVariable;

@Model(adaptables = SlingHttpServletRequest.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class PageMetaModel {

    @ScriptVariable
    private Page currentPage;

    public String getLabel() {
        return currentPage.getDepth() <= 2 ? "Home" : currentPage.getTitle();
    }
}
```

- [ ] Create `apps/academy/components/page-meta/` and wire the model with `data-sly-use`
- [ ] Render the value with a single expression: `${meta.label}` — no branching in the markup

**Acceptance:** the value comes from page context; the HTL is one expression.

### 4 — Read the value through a Sling Model

Java (provided — `core/.../models/notice/NoticeModel.java`):

```java
package biz.netcentric.academy.core.models.notice;

import org.apache.sling.api.resource.Resource;
import org.apache.sling.models.annotations.Default;
import org.apache.sling.models.annotations.DefaultInjectionStrategy;
import org.apache.sling.models.annotations.Model;
import org.apache.sling.models.annotations.injectorspecific.ValueMapValue;

@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class NoticeModel {

    @ValueMapValue
    @Default(values = "Heads up!")
    private String message;

    public String getMessage() {
        return message;
    }
}
```

- [ ] Swap the HTL from `${properties.message}` to the model: `data-sly-use.notice="...NoticeModel"` → `${notice.message}`
- [ ] Rebuild & deploy `core`, then clear the dialog field and confirm the default shows

**Acceptance:** an empty field falls back to the default; `data-sly-use` resolves the model.

### 11 — Build a complete "Promo" component

Java (provided — `core/.../models/promo/PromoModel.java`):

```java
package biz.netcentric.academy.core.models.promo;

import org.apache.sling.api.resource.Resource;
import org.apache.sling.models.annotations.Default;
import org.apache.sling.models.annotations.DefaultInjectionStrategy;
import org.apache.sling.models.annotations.Model;
import org.apache.sling.models.annotations.injectorspecific.ValueMapValue;

@Model(adaptables = Resource.class,
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class PromoModel {

    @ValueMapValue @Default(values = "") private String title;
    @ValueMapValue @Default(values = "") private String text;
    @ValueMapValue @Default(values = "") private String ctaLabel;
    @ValueMapValue @Default(values = "") private String ctaUrl;

    public String getTitle()    { return title; }
    public String getText()     { return text; }
    public String getCtaLabel() { return ctaLabel; }
    public String getCtaUrl()   { return ctaUrl; }
}
```

Combine everything into one new component, `apps/academy/components/promo/`:

- [ ] Dialog: title, body text, CTA label, and CTA link (`pathfield`)
- [ ] HTL wires `PromoModel` with `data-sly-use` and stays a thin view
- [ ] Clientlib in BEM, loaded by category; one loader behaviour if it needs JS
- [ ] Allowed by policy, one design-dialog option, and two Style System variations

**Acceptance:** an author can insert, fill, restyle, and configure the component end to end without code changes.

### 12 — (Stretch) Extend a Core Component

Java (provided — `core/.../models/title/TitleImpl.java`):

```java
package biz.netcentric.academy.core.models.title;

import com.adobe.cq.wcm.core.components.models.Title;
import org.apache.sling.api.SlingHttpServletRequest;
import org.apache.sling.models.annotations.DefaultInjectionStrategy;
import org.apache.sling.models.annotations.Model;
import org.apache.sling.models.annotations.Via;
import org.apache.sling.models.annotations.injectorspecific.Self;
import org.apache.sling.models.annotations.via.ResourceSuperType;

@Model(adaptables = SlingHttpServletRequest.class,
       adapters = Title.class,
       resourceType = "academy/components/title",
       defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL)
public class TitleImpl implements Title {

    @Self @Via(type = ResourceSuperType.class)
    private Title delegate;

    @Override
    public String getText() {
        return delegate.getText() + " — Academy";   // the one override
    }

    // forward everything else to `delegate` (let your IDE generate these)
}
```

- [ ] Create `apps/academy/components/title/.content.xml` with `sling:resourceSuperType="core/wcm/components/title/v3/title"`
- [ ] Confirm it renders with the inherited dialog + HTL, then deploy the provided model

**Acceptance:** the component reuses the Core Component, changes one thing, and still exports the standard JSON.

<Callout type="do">Work top to bottom. For each setting, decide deliberately: per-instance content goes in the **dialog**, per-template configuration goes in the **policy**, and purely visual variations go in the **Style System**.</Callout>
